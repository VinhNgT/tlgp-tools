This workflow guides the end-to-end process of creating a TLGP screen specification Google Doc from annotated screenshots. It uses the `tlgp-annotation-tool` desktop tool for annotation, computer vision for element analysis, codebase cross-referencing for API discovery, and Google Docs MCP tools for document generation.

> [!IMPORTANT]
> **Reference Design Match Requirement**: All generated screen specifications must match the layout structure, fonts, colors, and overall formatting of the reference document exactly:
> [TLGP Reference Document](https://docs.google.com/document/d/1Sp9Oq7L0wyOGqg1Y7AAL9fOgB4efZgZ68NrlrbHonV8/edit?tab=t.0)

> [!WARNING]
> **Google Docs Communication Protocol & Browser Restriction**: You must exclusively use the `google-docs` MCP server tools to interact with, view, and update Google Documents. Do not open a web browser (e.g. via browser subagent or other browser tools) to read, view, or edit the Google Document, and do not attempt to bypass the MCP server by writing custom scripts (Python, Node.js, etc.) that connect directly to the Google Docs API.
> 
> **If MCP communication is not possible, you must pause immediately and ask the user for advice.**

All output must be in **Vietnamese**.

## Appendix A: Design Specification

All content must match the following styling exactly. Read this appendix in full before executing any phase.

### A.1 Global Styles

| Property | Value |
|---|---|
| Font | `Times New Roman`, `12pt` |
| Heading color | `#4F81BD` (RGB 0.31, 0.51, 0.74) |

### A.2 Heading Hierarchy & Markdown Layout Rules

To ensure the generated document layout matches the reference exactly, you must follow these precise markdown syntax and structural rules:

#### 1. General Document Structure & Formatting
* **Use Annotated Screenshots Only**: You must exclusively insert the annotated screenshot images generated by the `tlgp-annotation-tool` (e.g., `<screen_name>_annotated.png`, `<screen_name>_<id>_annotated.png`, etc.) into the Google Document. Do not insert raw, unannotated screenshots under any circumstances.
* **No Document Title or Description**: The document must NOT contain any overall document header, title (e.g., `# Đặc tả màn hình...`), or introduction text. It must start directly with the first component section (e.g., `### **1.1.1. Component...**`).
* **Line Spacing**: Ensure there is exactly one empty line before and after all headings, tables, and paragraphs to match the spacing of the reference document.
* **DTO Suffix Casing Match**: Ensure all DTO type names (e.g., `BannerDto`, `ExchangeRateDto`, `ProductListDTO`) exactly match the case used in the actual source code classes (some use `Dto` and some use `DTO` in the codebase).
* **Data-Specific Field and Text Precision**: All query parameter keys, request/response DTO field names, data types, and nullability status in the document tables must stay strictly true to the actual source code annotations and class properties. Never generalize, guess, or auto-format them.
* **API Response Formatting (Flat vs. Nested tables)**:
  * For list-wrapping responses like `ProductListDTO`, do NOT split them into separate DTO tables (like `ProductDTO fields` or `PageInfoDTO fields`). Instead, document them as a single flat table using dot notation and array brackets for child fields (e.g., `items[].product_id`, `items[].price.amount`, `page_info.has_next_page`, `page_info.end_cursor`).
  * Separate sub-DTO tables (like `PriceTiersDTO fields`) should only be used when they are complex nested lists/objects within a primary entity DTO (like `ProductDetailDTO`).
* **Primitive or List-only Responses**:
  * For APIs that return a primitive value or a simple list of primitives (e.g., `Response (data = int)` or `Response (data = List<int>)`), do NOT write a parameters table. Write a single-sentence textual description of the return value instead.

#### 2. Heading Levels & Syntax
* **Component Title**: Format exactly as `### **X.Y.Z. Component <name>**` (Named Style: `HEADING_3`, bold inside markdown stars). E.g., `### **1.1.1. Component tiêu đề màn hình chi tiết sản phẩm**`.
* **Sub-section Title**: Format exactly as `#### *X.Y.Z.N <title>*` (Named Style: `HEADING_4`, italic inside markdown stars). Ensure there is **NO trailing period** after the fourth digit in the numbering (e.g., `#### *1.1.1.1 Thông tin chung về chức năng*` instead of `1.1.1.1.`).
* **API Title**: Format as bold normal text: `**N. METHOD <description>**` (e.g., `**1. GET Chi tiết sản phẩm**`). Do NOT use markdown heading tags (like `##` or `###`).
* **API URL**: Format as plain (non-bold) normal text: `URL: <path>` (e.g., `URL: /mall-catalog/api/v1/...`). Do NOT bold the "URL: " prefix. The path must NOT be wrapped in backticks.
* **API Request Label**: Format as bold normal text: `**Request**` or `**Request Body (<DTOType>)**`. Do NOT use heading tags.
* **API Response Label**: Format as bold normal text: `**Response (data = <DTOType>)**` (e.g., `**Response (data = ProductDetailDTO)**`). Do NOT use heading tags.
* **Sub-DTO Table Title**: Format as regular, non-bold normal text with no heading tags (e.g., `PriceTiersDTO fields` or `StockUnitDTO fields (sku_list)`).

#### 3. Table Column Headers & Content Style
* **Component General Info Table (2x2)**:
  * Headers: Column 0 header must be exactly `Tên chức năng`. Column 1 header must be the component name itself (e.g., `| Tên chức năng | Component tiêu đề màn hình chi tiết sản phẩm |`).
  * Body Row: Row 1 must have `Mô tả` in Column 0, and the description in Column 1 (e.g., `| Mô tả | Mô tả tiêu đề màn hình chi tiết sản phẩm |`).
* **Screen-level Info Table (2x2)**:
  * Same structure as component info, but Row 0 Col 0 is `Tên màn hình` instead of `Tên chức năng`.
* **Main Screen General Info Table (N×2)**:
  * Header row: Row 0 must be `| Mô tả chức năng | <description> |` — this is the header (bold, bg `#F2F2F2`).
  * Data rows: Rows 1+ contain the remaining fields (e.g. `| Người thực hiện (Actor) | ... |`).
* **UI Elements Table (7 cols)**:
  * Headers: Column 1 header must be exactly `Tên` (NOT `Tên thành phần`). The columns are: `| STT | Tên | Loại Control | Bắt buộc | Độ dài tối đa | Chỉnh sửa | Mô tả |`.
* **Interaction Events Table (2 cols)**:
  * Headers: Headers must be exactly `| Hành động của tác nhân | Phản ứng của hệ thống |`.
  * Row Content: Action and reaction descriptions must NOT use numbered lists or formal prefixes like `1. Người dùng...` or `Hệ thống...`. They must be concise and direct: e.g., `Click vào nút Back` | `Quay về màn trước`.
  * **Omit No-Action Rows**: Do NOT include rows where the system reaction is "Không có hành động" or equivalent. Only document interactions that produce a meaningful system response.
  * **Screen Entry Events**: If entering the screen itself triggers a business operation (e.g., fetching data, calling an API, checking session status), document this as the first row in the table. The action should be `Khởi động màn hình` and the reaction should describe the triggered operation concisely (e.g., `Khởi động màn hình` | `Gọi API lấy thông tin chi tiết và hiển thị UI`).

| Content | Markdown Syntax | Named Style | Text Format |
|---|---|---|---|
| Component title | `### **X.Y.Z. Component <name>**` | `HEADING_3` | **bold**, color `#4F81BD` |
| Sub-section | `#### *X.Y.Z.N <title>*` | `HEADING_4` | *italic*, color `#4F81BD` |
| API heading | `**N. METHOD <description>**` | `NORMAL_TEXT` | **bold** (Do NOT use `##` or `###`) |
| API URL | `URL: <path>` | `NORMAL_TEXT` | plain text, non-bold (Do NOT bold the "URL: " prefix) |
| API Request label | `**Request**` or `**Request Body (<DTOType>)**` | `NORMAL_TEXT` | **bold** (Do NOT use headings) |
| API Response label | `**Response (data = <DTOType>)**` | `NORMAL_TEXT` | **bold** (Do NOT use headings) |
| Sub-DTO Table Title | `<DTOType> fields` or `<DTOType> fields (<field>)` | `NORMAL_TEXT` | regular, non-bold, no heading |

## Appendix B: Content Structure Reference

### B.1 Per-Component Section Template

For each **non-leaf** component at any nesting level (which has an associated cropped annotated image generated by `tlgp-annotation-tool`, in annotation order), write these sections. Leaf components (those with no children) are **excluded** — they do not get their own section and are only listed as rows in their parent's UI Elements table.

1. **Component heading**: `X.Y.Z. Component <label>` → `HEADING_3`
2. **Sub-section**: `X.Y.Z.1 Thông tin chung về chức năng` → `HEADING_4` + Info Table (2×2)
3. **Sub-section**: `X.Y.Z.2 Màn hình chức năng` → `HEADING_4` + annotated component image
   > [!IMPORTANT]
   > **Single Component Image Rule**: Each component section must have exactly one annotated component image. No additional images are allowed in a component section.
   > 
   > [!IMPORTANT]
   > **Handling Leaf Components & No Agent Image Splitting**:
   > Under no circumstances is the agent allowed to write, run, or execute any custom post-processing scripts (e.g. using PIL/Pillow) to crop, cut, or split images. Only the user is allowed to split or crop images (via the `tlgp-annotation-tool` tool). The agent must **exclusively** use the native images generated and exported directly by the user from `tlgp-annotation-tool`.
   > - Leaf components (nodes with no children/sub-components) will not have cropped annotated images generated by `tlgp-annotation-tool`.
   > - Because leaf components do not have separate cropped images, they **must not** have their own separate "Component" sections in the specification document. They are strictly described inside their parent's UI table.
4. **Sub-section**: `X.Y.Z.3 Mô tả chi tiết các thành phần trên màn hình` → `HEADING_4` + UI Elements Table (7 cols)
   > [!IMPORTANT]
   > **Matching Annotated Image IDs**:
   > The rows in the "Mô tả chi tiết các thành phần trên màn hình" (UI Elements) table must match the IDs on the annotated image exactly—no more, no less. Each annotated child ID must correspond to exactly one row in the table, and no extra rows should be added.
5. **Sub-section**: `X.Y.Z.4 Xử lý luồng sự kiện tương tác (N bước)` → `HEADING_4` + Interaction Events Table (2 cols)
   > [!IMPORTANT]
   > **Step Count Accuracy in Section Headings**:
   > In the sub-section heading `X.Y.Z.4 Xử lý luồng sự kiện tương tác (N bước)`, the number `N` must exactly match the number of steps/rows (excluding the header row) in the interaction events table (e.g. `(2 bước)` or `(0 bước)` if there are no events). Never hardcode a default value (such as `(3 bước)`) across headings.

### B.2 Screen Overview Section Template

After all component sections, write the full-screen overview:

1. **Screen heading**: `X.Y.M. Màn hình <screen_name>` → `HEADING_3` (M = total number of component sections + 1)
2. **Thông tin chung**: Screen General Info Table (N×2)
3. **Màn hình chức năng**: Root-level annotated image(s)
   > [!NOTE]
   > The screen section is the only section in the document that can contain multiple images (e.g., `_part1.png`, `_part2.png`, etc.) if the screenshot was split using cut lines.
4. **Mô tả chi tiết**: UI Elements Table (7 cols) listing all top-level annotation boxes. Non-leaf nodes (nodes with children) must have `Loại Control` = `Component`. Leaf nodes (nodes without children) must have their actual control type (e.g., `Button`, `Text`, `Image`) and must NOT be promoted to Component type.
5. **Xử lý luồng sự kiện**: Full-screen interaction table

### B.3 API Documentation Template

After the screen overview section:

1. **Heading**: `N. <HTTP_METHOD> <Vietnamese description>` → bold, normal text
2. **URL line**: `URL: <endpoint_path>` → plain text, non-bold
3. **Request section**: API Table (6 cols) or `Không có tham số`
4. **Response section**: API Table (6 cols)
5. **Sub-DTO tables**: Nested DTO tables as needed

## Appendix C: Export JSON & Image File Reference

### C.1 Export JSON Structure

The `tlgp-annotation-tool` exports JSON in this structure:

```jsonc
{
  "screen_name": "Chi tiết sản phẩm",
  "description": "Màn hình hiển thị thông tin chi tiết sản phẩm",
  "original_image": "/path/to/screenshot.png",
  "image_width": 1080,
  "image_height": 2400,
  "cut_lines": [800, 1600],  // optional
  "components": [
    {
      "id": 1,
      "label": "Tiêu đề màn hình",
      "bounds": { "x": 0, "y": 0, "w": 1080, "h": 200 },
      "children": [
        {
          "id": 1,
          "label": "Button back",
          "bounds": { "x": 10, "y": 10, "w": 40, "h": 40 }
          // bounds are relative to parent
        }
      ]
    }
  ]
}
```

Coordinates in JSON are relative to their parent. Root-level boxes are relative to (0, 0).

### C.2 Exported Image Files

| File Pattern | Content |
|---|---|
| `{name}_annotated.png` | Full screenshot with Level 0 boxes (no cuts) |
| `{name}_annotated_part{N}.png` | Segment N of split screenshot (with cuts) |
| `{name}_{parent_path}_annotated.png` | Cropped to parent box at any nesting path (e.g. `{name}_{id}_annotated.png`, `{name}_{id}_{child_id}_annotated.png`, etc.), showing its immediate children |

### C.3 Helper Scripts

Three helper scripts are available in the `scripts/` directory:

| Script | Purpose | When to use |
|---|---|---|
| `extract_placeholder_indices.py` | Parse `readDocument` JSON to find `[TABLE_*]` and `[IMAGE_*]` placeholder indices | Step 4B (before table insertion) |
| `extract_table_ranges.py` | Parse `readDocument` JSON to classify tables and extract text ranges for styling | Step 4D.1 (before styling) |
| `extract_heading_ranges.py` | Parse `readDocument` JSON to find exact text ranges and levels of H3/H4 headings | Step 4D.1 (before styling) |

Both scripts are pure local JSON parsers (zero API cost). Run with:
```bash
$env:PYTHONUTF8=1; uv run python scripts/<script_name>.py <json_file_path>
```

## Prerequisites

- Python 3.11+ and `uv` installed. Always use `uv run python` instead of raw `python` when executing python scripts or commands.
- **Python Console & File Encoding (Avoiding UnicodeEncodeError)**:
  - When running Python scripts that handle Vietnamese text (e.g., printing names containing Vietnamese characters like `Trang_chủ` or reading/writing files), Windows consoles defaulting to `cp1252` will throw a `UnicodeEncodeError`.
  - **Resolution / Prevention**:
    1. **Environment Variables (Best Practice)**: Set one of the following environment variables to force Python to use UTF-8:
       - **Force Global UTF-8 Mode (`PYTHONUTF8`)**:
         - **PowerShell**: `$env:PYTHONUTF8=1` (run this before your script, or prepend: `$env:PYTHONUTF8=1; uv run python script.py`)
         - **CMD**: `set PYTHONUTF8=1`
       - **Force Standard I/O Encoding (`PYTHONIOENCODING`)**:
         - **PowerShell**: `$env:PYTHONIOENCODING="utf-8"`
         - **CMD**: `set PYTHONIOENCODING=utf-8`
    2. **Windows Console Code Page Change**: Change the active code page to UTF-8 before running the command:
       - **CMD / PowerShell**: `chcp 65001`
    3. **Script Code Fixes**:
       - Reconfigure standard output/error encoding at the very start of the script:
         ```python
         import sys
         if sys.platform == "win32":
             sys.stdout.reconfigure(encoding='utf-8')
             sys.stderr.reconfigure(encoding='utf-8')
         ```
       - Always specify `encoding='utf-8'` when opening text files (e.g., `open(filepath, 'r', encoding='utf-8')`).
       - Set `ensure_ascii=False` when using `json.dumps()` or `json.dump()` with Vietnamese text.
- Access to the target screen's screenshots.
- Google Docs MCP server configured.
- The `tlgp-annotation-tool` tool.

## Phase 1: User Prepares Screenshots & Annotations

### Step 1.1: Capture Screenshots

1. Open the mobile app simulator or device.
2. Navigate to the target screen.
3. Capture screenshot(s) of the screen.
4. If the screen is scrollable, capture multiple overlapping screenshots of each segment.

### Step 1.2: Agent Launches `tlgp-annotation-tool`

**The agent must launch `tlgp-annotation-tool` for the user.** Use `run_command` to start the tool.

**Always use `-o` to pre-set the output directory** so you know where to find the exported files without asking the user:

```bash
# Recommended: launch with preset output directory
uv run python -m tlgp_annotation_tool -o ./agent-tmp
```

```bash
# With screenshot(s) pre-loaded + output directory
uv run python -m tlgp_annotation_tool /path/to/screenshot.png -o ./agent-tmp
```

- Working directory: the `tlgp-annotation-tool` package root
- The command opens a GUI window. Send it to the background (set `WaitMsBeforeAsync` to `2000`).
- Screenshots are **optional** on the command line — the tool has an "Open Screenshot..." button and also supports drag-and-drop.
- If multiple screenshots are provided via CLI, the tool stitches them vertically.
- **After launching, STOP calling tools.** Tell the user you've launched the tool and ask them to notify you when they've finished annotating and exporting. Do NOT poll or try to proceed until the user confirms they're done.

### Step 1.3: Annotate Level 0 (Major Components)

1. Ensure you are viewing the root level canvas.
2. Switch to **Draw** mode (shortcut `D`).
3. Draw red boxes around all major components on the screen (e.g., Title Bar, Image Banner, Product Info Section, Bottom Buttons).
4. The sidebar list will display the components. Double-click or press Enter to rename each component with a descriptive Vietnamese name.
5. Ensure components are in correct visual order (top-to-bottom). Drag-and-drop in the sidebar to reorder if necessary.

### Step 1.4: Annotate Level 1 (Sub-elements)

1. Select a Level 0 component in the sidebar.
2. Press **Enter** to drill into that component.
3. The canvas will transition to show only the cropped region of the selected component.
4. Draw boxes around individual UI controls inside this component (buttons, text labels, icons, images, etc.).
5. Add descriptive Vietnamese names for each sub-element.
6. Press **Escape** to return to Level 0, then repeat for each component.

### Step 1.5: Annotate Deeper Nested Levels (N-level support)

The tool supports arbitrary nesting depth. If any sub-component at Level 1 or below contains its own nested children:
1. Drill into the child box by selecting it and pressing **Enter**.
2. The canvas will focus on the cropped parent box region, showing any child boxes inside.
3. Draw nested boxes, rename them, and press **Enter** to go deeper if needed.
4. Press **Escape** to navigate one level back up.

### Step 1.6: Use Cut Lines for Long Screenshots (Optional)

If the screenshot is very tall:
1. Press **C** to open the Cut Line Editor.
2. Click **Add Cut** and place horizontal cut lines at logical breakpoints.
3. Click **OK** to commit. This will split the annotated export into multiple PNG parts.

### Step 1.7: Fill Screen Metadata & Export

1. Click **Screen Info** in the toolbar (or menu).
2. Fill in:
   - **Screen Name**: e.g., `Chi tiết sản phẩm`
   - **Description**: Brief description of the screen's purpose.
3. Press **Ctrl+S** (or File → Export).
4. Select an output directory. The tool creates a subfolder containing:
   - `<screen_name>.json` — All hierarchical annotations and metadata.
   - `<screen_name>_annotated.png` (or `_part1.png`, `_part2.png` if cuts exist) — Root-level annotated image(s).
   - `<screen_name>_<id>_annotated.png` — Cropped annotated images for each component with children.

## Phase 3: Agent Analysis (Vision + Codebase)

> [!IMPORTANT]
> **Annotated Images as the Source of Truth**:
> - The annotated screenshots/images are the absolute source of truth for the screen's layout, components, and hierarchy.
> - The source code is only used as a reference to discover routes, DTOs, and API details.
> - You must NOT modify, edit, or crop/cut the annotated screenshots or the hierarchy/data exported from the tool.
> - If you find any discrepancies between what is shown in the annotated screenshots and what is implemented in the source code (e.g., elements that exist in the screenshots but are missing in the code, or vice versa), you must warn the user of these discrepancies in your report or document rather than attempting to modify the screenshots/specifications to force a match.

### Step 3.1: Parse JSON & Infer Labels

1. Read the exported `.json` file to obtain: `screen_name`, `description`, `image_width`, `image_height`, `cut_lines` (if any), `components[]` tree.
2. **Auto-infer descriptive labels**: Scan all component and child labels. If any label is still generic (matching patterns like `Component \d+`, `Sub-component \d+`, or `Item \d+`):
   - View the corresponding annotated image (`view_file` on root and component-level annotated PNGs) to identify the actual UI element.
   - Infer a descriptive Vietnamese name from the visual content (e.g., a component showing a logo → `Logo`, a search bar → `Thanh tìm kiếm`, icons showing ⋮ | × → `Nút menu`, `Nút đóng`).
   - **Use the inferred names throughout the entire document** (headings, info tables, UI element tables, interaction tables) instead of the generic labels.
   - Do NOT ask the user to rename labels — infer them yourself from the visual content.

### Step 3.2: View Images & Classify Controls

### Step 3.3: Trace Codebase (Routes, Cubits, APIs, DTOs)

Search the project codebase to discover APIs and data models.

1. **Find Screen Route & Cubit**:
   - Search for the screen name in AutoRoute configuration (`grep_search` in `lib/**/router/` or `lib/**/routes/`).
   - Find the screen widget class and its associated Cubit (`grep_search` for `BlocProvider<...Cubit>`).
   - **If the screen file uses `part` directives** (e.g., `part 'private/..._private_screen.dart'`), also read the part files. The actual widget build logic, BlocBuilder/BlocListener usage, and Cubit interactions are typically in the `part` file, not the main screen file.

2. **Trace Use Cases & API Endpoints**:
   - Open the Cubit file and find `callApi` invocations to identify Use Cases.
   - Open each Use Case to find the Repository method it calls.
   - Open the Repository implementation to find the API Service method.
   - Open the API Service (Retrofit) to extract:
     - **HTTP Method**: Must match the Retrofit annotation (`@GET`, `@POST`, `@PUT`, `@DELETE`, etc.) exactly.
     - **Endpoint Path**: Must match the Retrofit annotation path exactly. Preserve any path parameters (e.g. `{productID}`).
     - **Request Parameters / Body DTO**:
       - Check the parameter annotations (e.g. `@Query('merchant_id') required String? merchantId`, `@Body() required FavoriteProductRequestDTO body`).
       - **Exact Names**: The parameter name in the document table must match the network key name from the annotation (e.g. `merchant_id` from `@Query('merchant_id')`, NOT the Dart variable `merchantId`).
     - **Response Type**: Note the exact DTO class returned (e.g. `UtilityDto` vs `ProductListDTO`). Note whether the return type is unwrapped (`@UnwrapData()`). If not unwrapped, the response envelope will wrap the data (e.g. `BaseResponseDTO<List<int>>` maps to response type `Response (data = List<int>)`).

3. **Extract DTO Fields**:
   - For each Request DTO and Response DTO:
     - Open the DTO file and read all `@JsonSerializable` fields.
     - **Determine Exact Field Name (Network Key)**:
       - If a field has a `@JsonKey(name: '...')` annotation, use the exact name specified in the `name` attribute.
       - If no `@JsonKey` annotation is present, check the class-level `@JsonSerializable` configuration. If `fieldRename: FieldRename.snake` is configured, convert the Dart camelCase name to snake_case (e.g. `productId` becomes `product_id`).
       - Never guess or change the field names in the tables; they must represent the exact JSON keys serialized/deserialized by the code.
     - **Determine Exact Dart Type & Nullability**:
       - Check the type declaration and nullability (e.g. `String?` is nullable, `String` is non-nullable).
       - In the API parameter/field tables, under **Kiểu dữ liệu**, write the exact type including the question mark for nullability (e.g., `String?`, `int?`, `double?`, `bool?`, `List<String>?`, `List<CategoryDto>?`).
       - For nested DTOs, recursively extract their fields using the same exact field name and type rules.
     - Map each field to a Vietnamese description based on the field name and context.

4. **Compile API Documentation**: For each API endpoint, prepare: sequential number and Vietnamese title, URL path, request parameters table, response fields table, and sub-DTO tables as needed.

## Phase 4: Google Doc Generation

This phase is broken into 4 sequential sub-tasks. Complete each sub-task fully before moving to the next.

> [!WARNING]
> **API Rate Limit & Latency Mitigation**: Phase 4 generates ~460 API calls for a typical document. To avoid rate limit errors:
> 1. Process write operations in batches of up to **35 calls**.
> 2. If blocked by a rate limit error (quota exceeded), wait **30 seconds** before retrying.
> 3. Show a progress percentage (e.g., "Progress: 44% (Batch 4/9) - 130/293 calls completed") in each turn response.

### Sub-task 4A: Create Document & Insert All Text Content (1 API call)

Use `createDocument` with `initialContent` to create the doc and insert all text in a **single call**:

```json
// MCP call: google-docs / createDocument
{
  "title": "<screen_name>",
  "initialContent": "### **1.1.1. Component...**\n#### *1.1.1.1 Thông tin...*\n[TABLE_INFO_1]\n...",
  "contentFormat": "markdown"
}
```

Build the markdown string from the Component Analysis Records (Step 3.4). Use these placeholder markers where tables and images will be inserted later:

| Placeholder | Table Type | Data Source |
|---|---|---|
| `[TABLE_INFO_N]` | Info Table 2×2 | Component N label + description |
| `[TABLE_SCREEN_INFO]` | Screen General Info N×2 | screen.* fields |
| `[TABLE_UI_N]` | UI Elements 7-col | Component N children |
| `[TABLE_INTERACTION_N]` | Interaction 2-col | Component N interactions |
| `[TABLE_API_REQ_N]` | API Request 6-col | apis[N].requestParams |
| `[TABLE_API_RES_N]` | API Response 6-col | apis[N].responseFields |
| `[TABLE_API_SUB_N_M]` | Sub-DTO 6-col | apis[N].subDtos[M] |
| `[IMAGE_N]` | Annotated image | Component N imagePath |
| `[IMAGE_SCREEN_N]` | Root screenshot | screen.imagePaths[N] |

### Sub-task 4B: Insert Tables (Descending Index Order)

To avoid character index shifting bugs, all insertions must be performed in **descending character index order** (bottom-to-top).

**Steps**:

1. `readDocument(format='json')` → save to temp file (`agent-tmp/doc_structure.json`) — 1 API call
2. Run `extract_placeholder_indices.py` on the temp file — 0 API calls
3. The script outputs all placeholder indices sorted descending
4. For each placeholder (bottom-to-top):
   a. `deleteRange` to remove the placeholder text — 1 call
   b. `insertTableWithData` at the placeholder's `startIndex` — 1 call
   
   Set `hasHeaderRow: true` for **ALL** table types (info, screenInfo, uiElements, interaction, api). Every table has a standard header Row 0.

```json
// MCP call: google-docs / insertTableWithData
{
  "documentId": "<docId>",
  "index": <startIndex>,
  "hasHeaderRow": true,
  "data": [
    ["STT", "Tên", "Loại Control", "Bắt buộc", "Độ dài tối đa", "Chỉnh sửa", "Mô tả"],
    ["1", "Button back", "Icon", "", "", "", "Nút quay lại"]
  ]
}
```

5. After all tables are inserted, call `listDocumentTables` — 1 API call
6. Classify tables and build the **Table Type Registry**:

**Classification rules** (deterministic):
- 7 cols → `uiElements`
- 6 cols → `api`
- 2 cols → call `getTableStructure` and check Row 0 Col 0 text:
  - `"Tên chức năng"` or `"Tên màn hình"` → `info`
  - `"Mô tả chức năng"` → `screenInfo`
  - `"Hành động của tác nhân"` → `interaction`

> [!IMPORTANT]
> Row count alone is **NOT** sufficient for classifying 2-col tables. Interaction tables can have as few as 2 rows (1 header + 1 data), which is the same as Info tables. Always check cell text content.

7. Save the Table Type Registry locally:
```jsonc
{
  "info": ["table:body:0", "table:body:3", ...],
  "screenInfo": ["table:body:24"],
  "uiElements": ["table:body:1", "table:body:4", ...],
  "interaction": ["table:body:2", "table:body:5", ...],
  "api": ["table:body:27", "table:body:28", ...]
}
```

### Sub-task 4C: Insert Images (Descending Index Order)

After all tables have been inserted, perform a **fresh `readDocument`** call, then insert images in **descending character index order** (bottom-to-top).

> [!IMPORTANT]
> **Complete Image Utilization**: You must insert and utilize every single annotated screenshot image generated and exported by the `tlgp-annotation-tool` (including all root-level screenshot parts and all component-level cropped images). No generated annotated images may be omitted or left unused in the Google Document.

> [!IMPORTANT]
> **One Image Per Component**: Each component section in the document must contain exactly one annotated image representing that component. The screen overview section is the only section that can contain multiple images (e.g., when the root screen is split into multiple parts like `_part1.png`, `_part2.png`, etc., due to cut lines).

> [!WARNING]
> **Path Sandboxing Restriction**: The Google Docs MCP server requires all `localImagePath` arguments to be located within the active working directory of the MCP server process (which is the IDE program installation directory, e.g., `C:\Users\Victor\AppData\Local\Programs\Antigravity IDE\`).
> 
> To bypass this restriction:
> 1. Copy the images from the workspace to the IDE program directory: `C:\Users\Victor\AppData\Local\Programs\Antigravity IDE\`.
> 2. Pass the absolute path under the program directory to `insertImage` (e.g., `C:/Users/Victor/AppData/Local/Programs/Antigravity IDE/<image>.png`).
> 3. Once insertion succeeds (the image has been uploaded to Google Drive and placed in the document), delete the copied files from the program directory to keep it clean.

**Steps**:

1. `readDocument(format='json')` → save to temp file — 1 API call
2. Find image placeholder indices (from the JSON)
3. Copy images to IDE program directory. **Vietnamese filename workaround**: When copying files with Vietnamese characters (e.g., `Trang_chủ_2_annotated.png`), rename them to ASCII-safe equivalents during copy (e.g., `Trang_chu_2_annotated.png`) to avoid path encoding issues.
4. For each placeholder (bottom-to-top):
   a. `deleteRange` to remove placeholder text — 1 call
   b. `insertImage` at the index — 1 call

```json
// MCP call: google-docs / insertImage
{
  "documentId": "<docId>",
  "index": <startIndex>,
  "localImagePath": "C:/Users/Victor/AppData/Local/Programs/Antigravity IDE/<image>.png",
  "width": 468
}
```

Use `width: 468` (6.5 inches = standard page width minus margins) for **root-level annotated screenshots** only. For **component-level cropped images**, omit the `width` parameter to let them render at their natural/proportional size.

5. Delete copied images from IDE directory

### Sub-task 4D: Apply Styling (Ordered 5-Step Pipeline)

Execute these 5 steps in exact order. Do not skip or reorder steps.

#### Step 4D.1: Read Document Structure (1 API call + 2 scripts)

#### Step 4D.2: Baseline Font + All Heading Styles

> [!WARNING]

3. **For each H4 heading** — 1 `applyTextStyle` call per heading (italic + color):

// applyTextStyle — use startIndex/endIndex from the heading check script output
