1. **Vietnamese Translation Rule:** All component labels, UI descriptions, actions, reactions, and discrepancies written into the analysis JSON structure must be in Vietnamese.
2. **Strict Read-Only Mode:** You are strictly prohibited from mutating the Annotator state directly. All active mutations must be performed by the user in the GUI.
3. **No Direct REST API Bypass:** Do NOT bypass the MCP server layer (e.g. using curl, terminal/shell commands, or custom scripts) to interact directly with the running Annotator REST API.
4. **Strict Validation Pipeline:** Always run `generate_spec_doc(validate_only=True)` and review/resolve all validation warnings and errors before calling the tool with `validate_only=False`.
5. **Analysis JSON and Workspace Archive Record-Keeping:** When compiling a specification document (`validate_only=False`), the tool automatically saves the final analysis JSON data as `analysis.json` and exports the current workspace backup as `workspace.zip` in the same directory as the generated `.docx` file.
6. **Annotated vs Raw Image Separation:** The `export_images` tool exports **both** annotated images (with bounding box overlays and labels) and raw images (clean crops without overlays) in a single call. The output directory contains an `annotated/` subdirectory and a `raw/` subdirectory. Use annotated images (from `annotated/`) as the `imageFile` references in the analysis JSON for inclusion in the generated DOCX. Use raw images (from `raw/`) for AI vision analysis so that bounding box overlays do not obscure visual details or interfere with classification.
7. **DFS Document Ordering:** You must document the components in the `components` list in post-order Depth-First Search (DFS) sequence (child components first, parent components later).
8. **Non-Leaf Box Completeness:** Every single non-leaf box (any component box that contains child elements/boxes in the workspace state) must have a dedicated component entry in `components` in the analysis JSON, with `isLeaf` set to `false`.
9. **Leaf Components Vision & Context Analysis:** While only the root screen and non-leaf components (`isLeaf: false`) receive dedicated sections in the generated DOCX, you must still analyze all leaf components (`isLeaf: true`) during vision and codebase analysis. Analyzing leaf components is required to gain a comprehensive understanding of layout context, parent component behavior, and API integrations.
10. **API Placement (Using vs. Calling) & Uniqueness:** Document each API under the specific non-leaf component that **uses** the API's response data to render/update the UI or handle the action, NOT where the API request is called/triggered in the code (e.g., if a state manager or screen coordinator triggers the call but a sub-component displays the data, place the API in that sub-component). Only place an API under the Screen Overview if it is directly used at the root level without being delegated to any non-leaf component.
    **Common Ancestor Placement:** Because a single API documentation block must belong to exactly one component or screen, if an API is used by multiple child components (e.g., different sub-components display different fields returned by the same API), you must place the API documentation on the closest parent/ancestor component (or the screen overview) that encompasses all the child components that use it.
    **No Duplicates Allowed:** Duplicate APIs (either having the same sequential `number`, or having the same `(method, url)` combination) are strictly prohibited across all sections of the document and will fail validation.
11. **Target Codebase Mismatch Check (if source code is available):** If you have access to the project's source code, cross-reference the screenshot and components in the workspace with the codebase. If they do not match (e.g., outdated screenshot, missing UI elements in code), stop and ask the user for guidance before proceeding.
12. **Vision-Derived Naming & Accurate Descriptions:** All user-entered strings in the workspace state — component labels, screen name, and screen description — are rough **suggestions only**. When building the analysis JSON, you must **not** copy them verbatim if they are generic (e.g., `"Component"`, `"Thành phần 1"`), empty, or low-quality. Instead:
    - **Component labels:** Analyze the **raw** cropped image for each component and assign a precise, descriptive Vietnamese label that reflects what the UI element actually is (e.g., `"Thanh điều hướng"`, `"Phần tìm kiếm sản phẩm"`, `"Danh mục nổi bật"`).
    - **Component descriptions:** Each component's `description` must accurately describe the **specific visual content** within that component's annotated bounding box. Inspect the cropped image and describe what is actually visible — the layout, the elements it contains, and its purpose. Do not write generic filler like `"Thành phần hiển thị nội dung"`.
    - **Screen name & description:** Derive from the overall purpose of the screen visible in the full screenshot. Do not leave empty or use placeholder text.
