1. **Vietnamese Translation Rule:** All component labels, UI descriptions, actions, reactions, and discrepancies written into the analysis JSON structure must be in Vietnamese.
2. **Strict Read-Only Mode:** You are strictly prohibited from mutating the Engine state directly. All active mutations must be performed by the user in the GUI.
3. **No Direct REST API Bypass:** Do NOT bypass the MCP server layer (e.g. using curl, terminal/shell commands, or custom scripts) to interact directly with the running Engine REST API.
4. **Parameter Payload Limitation:** Direct `analysis` dictionary arguments to `generate_spec_doc` must not exceed 10KB to prevent JSON-RPC transport middleware corruption. For larger payloads, save the data using the `write_analysis_json` tool first, and pass the resulting absolute path via `analysis_path`.
5. **Strict Validation Pipeline:** Always run `generate_spec_doc(validate_only=True)` and review/resolve all validation warnings and errors before calling the tool with `validate_only=False`.
6. **Analysis JSON and Workspace Archive Record-Keeping:** When compiling a specification document (`validate_only=False`), the tool automatically saves the final analysis JSON data as `analysis.json` and exports the current workspace backup as `workspace.zip` in the same directory as the generated `.docx` file.
7. **Children Annotations Overlay & Vision Separation:** You must use images with children annotations enabled (bounding boxes and labels visible) inside the final generated DOCX (by downloading them to your `exportDir` with `show_root_children=True` and `show_component_children=True`). However, for **AI vision analysis**, you must always inspect unannotated images (downloaded with `show_root_children=False` and `show_component_children=False`) so that bounding box overlays do not obscure visual details or interfere with classification.
8. **DFS Document Ordering:** You must document the components in the `components` list in post-order Depth-First Search (DFS) sequence (child components first, parent components later).
9. **Non-Leaf Box Completeness:** Every single non-leaf box (any component box that contains child elements/boxes in the workspace state) must have a dedicated component entry in `components` in the analysis JSON, with `isLeaf` set to `false`.
10. **Leaf Components Vision & Context Analysis:** While only the root screen and non-leaf components (`isLeaf: false`) receive dedicated sections in the generated DOCX, you must still analyze all leaf components (`isLeaf: true`) during vision and codebase analysis. Analyzing leaf components is required to gain a comprehensive understanding of layout context, parent component behavior, and API integrations.
11. **API Placement (Using vs. Calling) & Uniqueness:** Document each API under the specific non-leaf component that **uses** the API's response data to render/update the UI or handle the action, NOT where the API request is called/triggered in the code (e.g., if a Cubit or Screen coordinator triggers the call but a sub-component displays the data, place the API in that sub-component). Only place an API under the Screen Overview if it is directly used at the root level without being delegated to any non-leaf component.
    **Common Ancestor Placement:** Because a single API documentation block must belong to exactly one component or screen, if an API is used by multiple child components (e.g., different sub-components display different fields returned by the same API), you must place the API documentation on the closest parent/ancestor component (or the screen overview) that encompasses all the child components that use it.
    **No Duplicates Allowed:** Duplicate APIs (either having the same sequential `number`, or having the same `(method, url)` combination) are strictly prohibited across all sections of the document and will fail validation.
12. **Target Codebase Mismatch Check:** You must always cross-reference the screenshot and components in the workspace with the target codebase. If the screenshot or components do not match the target codebase (e.g. outdated screenshot, mismatch in screen/component structure, or UI elements that cannot be found in the current codebase), you MUST stop and ask the user for guidance before proceeding with the analysis or document generation.
